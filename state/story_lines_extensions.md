# Story Lines Extensions: рейтинг, соцсети и ближайшие сюжетные крючки

Этот файл расширяет `state/story_lines.json` без создания нового основного state-файла.

Главная память всё равно хранится в:

```txt
state/story_lines.json
```

Этот файл объясняет, какие дополнительные блоки и линии нужно поддерживать внутри `story_lines.json`, если они ещё не добавлены в шаблон.

## line_rating

В `state/story_lines.json.lines` должна быть или может быть создана линия:

```txt
line_rating
```

Назначение:

- рейтинг Академии;
- классы и допуски;
- результаты проверок;
- доступ к тренировочным зонам;
- последствия спаррингов, стрельбищ, техники, контроля энергии;
- реакция инструкторов на результаты.

Пример структуры:

```json
{
  "title": "Рейтинг и классы Академии",
  "character_ids": [],
  "related_ids": ["akira", "livia_cross", "haru_foster", "raiden_sterling", "veronica_ellard", "kael_north"],
  "current_summary": "",
  "events": [],
  "rating_updates": [],
  "class_updates": [],
  "exam_results": [],
  "training_access": [],
  "open_threads": [],
  "compacted_summary": "",
  "compacted_history": []
}
```

## line_social_media_rumors

Соцсети и слухи — фон академии, а не главный сюжет.

Их нужно хранить коротко:

- что появилось;
- где появилось;
- кто мог увидеть;
- насколько сильно влияет на репутацию;
- вернётся ли это позже.

Соцсети не должны каждый раз превращаться в центральную сцену, занятие, расследование или моральный разбор.

Можно использовать поле:

```txt
daily_background_ping
```

Для коротких напоминаний вроде:

```txt
17 августа — в закрытом канале коротко обсуждают белые волосы Акиры и видео с площадки; пока это фон, не основная сцена.
```

## next_beats

В `state/story_lines.json` должен быть или может быть создан блок:

```txt
next_beats
```

Это мини-план ближайших сюжетных крючков.

Он нужен, чтобы ИИ не импровизировал каждый ход с нуля и не забывал, что завтра/через день кто-то должен подойти, долг должен продолжиться, слух должен всплыть, проверка должна иметь последствия.

Пример структуры:

```json
{
  "purpose": "Мини-план ближайших сюжетных крючков. Использовать как напоминание, а не как жёсткий сценарий.",
  "rules": {
    "keep_3_to_7_active_beats": true,
    "each_beat_has_date_or_condition": true,
    "do_not_force_if_scene_context_changed": true,
    "if_due_then_show_or_explain_delay": true,
    "remove_or_compact_after_resolved": true
  },
  "active": [
    {
      "beat_id": "haru_next_coffee_1198_08_18",
      "date_or_condition": "1198-08-18 morning / breakfast if Haru has access",
      "related_ids": ["akira", "haru_foster"],
      "summary": "Хару должен продолжить долг по кофе, если день и сцена логично позволяют.",
      "priority": "medium",
      "status": "open"
    }
  ],
  "resolved": []
}
```

## Как использовать next_beats

ИИ обязан перед сценой смотреть активные ближайшие крючки.

Если крючок подошёл по дате/условию, ИИ должен:

- показать его сценой;
- или коротко объяснить, почему он сорвался/перенёсся;
- или оставить открытым, если сцена логично не дала ему случиться.

Нельзя заставлять крючок сработать насильно, если текущая сцена ушла в другой важный конфликт.

Но нельзя и забывать его без объяснения.

## Минимум записей

Держать 3–7 активных next_beats.

Примеры:

- Хару должен продолжить долг по кофе;
- Киара может снова появиться рядом с Райденом и Хару;
- Вероника должна отреагировать на рейтинг/проверку Акиры;
- Ливия держит раздражение из-за Киары;
- Райден/Хару должны пересечься в спортивной или тренировочной зоне;
- соцсети коротко напоминают о событии, но не становятся центром главы.

## Что нельзя

Нельзя превращать next_beats в жёсткий сценарий.

Нельзя игнорировать действие игрока ради заранее написанного плана.

Нельзя держать закрытый beat в active.

Нельзя создавать отдельный файл на каждый beat.

Нельзя использовать next_beats, чтобы вытеснить основные события сцены.
