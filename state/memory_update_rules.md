# Memory Update Rules

Этот файл задаёт правила обновления памяти после каждой сцены.

Цель: чтобы предыстория не разваливалась, NPC не забывали события, отношения росли постепенно, а скрытый лор не попадал тем, кто его не знает.

## Главное правило памяти сессии

После каждой сцены нужно сохранять короткую память сессии.

ИИ не должен полагаться только на полный текст прошлых сцен или память чата.

Если в сцене появился важный NPC, начался конфликт, прозвучало имя, изменилась репутация, кто-то что-то увидел/услышал/узнал — это должно быть кратко сохранено.

Если событие не сохранено, последующие сцены начинают ломаться: NPC снова вводятся как новые, персонажи забывают знакомства, имена раскрываются повторно, знания путаются.

## Что обновлять после сцены

После каждой значимой сцены обновлять только то, что реально изменилось.

Возможные блоки:

- `current_state`
- `relationships`
- `reputation_state`
- `power_state`
- `rumors_state`
- `npc_registry`
- `location_registry`
- `inventory_state`
- `knowledge_state`
- `scene_history`
- `session_summary`
- `future_locks_progress`

Не обновлять всё подряд пустыми изменениями.

## session_summary

`session_summary` — короткая выжимка того, что уже произошло в текущей сессии.

Её цель — чтобы ИИ не вводил повторно уже появившихся NPC и не забывал, кто кого видел/знает.

После сцены добавлять короткий блок:

```json
{
  "scene_id": "auto_or_short_id",
  "date": "1198-08-15",
  "time": "после обеда",
  "location": "главный двор Академии",
  "participants": ["akira", "livia_cross", "npc_mika_veyl"],
  "summary": "Акира и Ливия пришли во двор. Мика Вейл задела тему белых волос Акиры и представилась.",
  "names_revealed_to_akira": ["Мика Вейл"],
  "seen_by_akira": ["npc_mika_veyl"],
  "akira_knows_now": ["Мика Вейл учится на первом курсе и любит цепляться к новичкам"],
  "npc_knowledge_changes": {
    "npc_mika_veyl": ["видела Акиру и Ливию во дворе", "знает, что у Акиры белые волосы"]
  },
  "open_threads": ["Мика может снова задеть Акиру из-за волос"],
  "do_not_reintroduce_as_new": ["npc_mika_veyl"]
}
```

Сохранять коротко, не полный пересказ.

Если нет отдельного файла `session_summary.json`, добавлять эти сведения в `scene_history` и notes.

## Защита от повторного ввода NPC

Перед созданием нового NPC ИИ обязан проверить:

- `characters/npc/npc_registry.md`;
- `scene_history`;
- `session_summary`, если есть;
- `knowledge_state`;
- `relationships`;
- `rumors_state`.

Если NPC уже появлялся, нельзя вводить его как нового.

Нельзя писать:

- “незнакомая девушка”;
- “новая студентка”;
- “какая-то девушка впервые подошла”;

если этот NPC уже был в сценах и Акира его видела/слышала/знает имя.

Правильно:

- если Акира знает имя — использовать имя;
- если Акира видела NPC, но имени не знает — использовать устойчивый дескриптор: `девушка с серебряной заколкой`, `парень с разбитой бровью`;
- если NPC уже общался с Акирой — учитывать прошлую сцену;
- если NPC уже конфликтовал/флиртовал/помогал — не обнулять отношение.

## Реестр NPC

ИИ может придумать NPC в сцене, но фиксировать только значимых.

Добавлять NPC в реестр, если он:

- получил имя и может вернуться;
- вступил в конфликт;
- стал объектом флирта/ревности;
- связан с важной группой;
- заметил важное;
- повлиял на репутацию;
- дал улику;
- связан с локацией;
- может стать ключевым;
- выглядит как потенциальная подруга, соперница, поклонница, враг или союзник.

Фоновый NPC без имени не добавляется.

Для значимого NPC сразу фиксировать:

```json
{
  "id": "npc_firstname_lastname_or_marker",
  "name_ru": "Имя Фамилия",
  "known_name_to_akira": true,
  "first_seen_scene": "1198-08-15 главный двор",
  "stable_descriptor": "девушка с серебряной заколкой",
  "role": "потенциальная подруга / соперница / студентка первого курса",
  "appearance_short": "невысокая, короткие тёмные волосы, серебряная заколка",
  "personality_short": "цепкая, язвительная, быстро оценивает чужую слабость",
  "energy_short": "не раскрыта",
  "knows_about_akira": ["видела белые волосы", "слышала имя Акиры, если оно прозвучало"],
  "relationship_to_akira": "настороженный интерес",
  "do_not_reintroduce_as_new": true
}
```

ID нельзя менять после фиксации.

Если имя ещё не известно Акире, `known_name_to_akira` должен быть false, но stable_descriptor обязателен.

## current_state

Фиксировать:

- дату;
- время;
- место;
- погоду, если она важна;
- состояние Акиры;
- активных персонажей;
- nearby персонажей;
- текущую сценовую цель;
- последний ввод игрока.

Время не должно двигаться по минутам без причины.

Если сцена была короткой, время может остаться тем же.

Если был разговор/переход/проверка — время должно сдвинуться логично.

## relationships

Отношения обновлять постепенно, но по направлению будущего канона.

Для пар персонажей можно вести показатели:

- `affection` — симпатия;
- `trust` — доверие;
- `tension` — напряжение;
- `jealousy` — ревность;
- `respect` — уважение;
- `fear` — страх;
- `curiosity` — интерес;
- `resentment` — обида.

Пример:

```json
{
  "akira__haru_foster": {
    "affection": 8,
    "trust": 3,
    "tension": 1,
    "jealousy": 0,
    "notes": ["Хару познакомился с Акирой в день прибытия и явно заинтересовался."]
  }
}
```

Не делать резких скачков без сильной сцены.

Например, один взгляд не даёт +50 любви.

## reputation_state

Репутация может быть разной в разных группах:

- новички;
- старшие курсы;
- инструкторы;
- поклонницы Хару;
- люди вокруг Райдена;
- комиссия;
- медицинский блок;
- группа Самуэля.

Фиксировать:

- кто что заметил;
- какие слухи пошли;
- кто уважает;
- кто презирает;
- кто боится;
- кто заинтересован.

## power_state

Фиксировать отдельно:

- публично показанный уровень;
- реальный уровень, если известен движку;
- кто видел проявление;
- кто понял неправильно;
- кто начал подозревать;
- последствия перегруза;
- ограничения контроля.

Для Акиры особенно важно:

- она скрывает способности и масштаб;
- она стабильна;
- она не машина убийства;
- она не раскрывает природу слишком рано.

## rumors_state

Слухи должны иметь источник и область распространения.

Пример:

```json
{
  "rumor_id": "akira_too_calm_on_check",
  "text": "На проверке Акира была слишком спокойной для новичка.",
  "source": "students_after_first_check",
  "spread": ["novice_dormitory", "academy_cafeteria"],
  "known_by": ["some_novices"],
  "truth_level": "partial"
}
```

Слухи не должны мгновенно знать все.

## location_registry

Если новая локация важна или повторяется, добавить:

- ID;
- название;
- тип;
- доступ;
- описание;
- что там есть;
- что там запрещено;
- сюжетное значение;
- что уже произошло.

Не перестраивать уже закреплённые локации без причины.

## inventory_state

Фиксировать только реальные предметы.

Отдельно:

- видимые предметы;
- предметы рядом;
- предметы в комнате;
- выданные академией предметы;
- скрытые предметы в кармане пространства;
- предметы, которые Акира достала публично;
- кто мог это заметить.

`current_state.visible_inventory` и `inventory_state.akira.visible_inventory` должны совпадать по видимым предметам.

`current_state.nearby_items` и `inventory_state.akira.nearby_items` должны совпадать по предметам рядом.

Если телефона нет, не писать это каждый раз.

Если предмет не был закреплён, не доставать его удобно из ниоткуда.

## knowledge_state

Самое важное правило: персонажи не получают знания автоматически.

Фиксировать:

- кто видел событие;
- кто слышал фразу;
- кому рассказали;
- кто сделал вывод;
- кто ошибочно понял;
- кто не знает;
- чьё имя Акира уже знает;
- кого Акира видела, но имени не знает.

Пример:

```json
{
  "akira": {
    "knows": ["Хару явно знаком с Райденом", "имя Хару Фостер"],
    "suspects": ["Райден избегает общения не из робости"],
    "seen_but_name_unknown": ["высокий тёмноволосый парень у стены"],
    "does_not_know": ["Райден гибрид", "связь через цикл", "имя Райдена, если оно ещё не прозвучало"]
  }
}
```

Hidden lore нельзя добавлять в `knows`, если персонаж не узнал это сценой.

## scene_history

Сохранять кратко:

- дата;
- время;
- локация;
- участники;
- действие игрока;
- итог сцены;
- важные реплики;
- изменения отношений;
- новые предметы;
- новые NPC;
- имена, которые прозвучали при Акире;
- NPC, которых Акира видела без имени;
- новые слухи;
- что нужно помнить в следующей сцене.

Не сохранять огромный полный текст сцены в каждую краткую память, если есть отдельное хранилище полной сцены.

## future_locks_progress

Фиксировать продвижение к будущему канону:

- Хару начал влюбляться в Акиру;
- Ливия заинтересовалась Райденом;
- Райден начал раздражённо замечать Акиру;
- Самуэль заметил Акиру;
- появились первые подозрения о скрытой силе;
- отношения Хару и Акиры стали заметны другим;
- Райден проявил ревность, но не признал её;
- Ливия переключилась на Хару после отказа Райдена;
- Хару получил причину позже отступить.

Эти пункты являются направлением канона, но каждый должен появляться через сцены и последствия, а не одним резким утверждением.

## Проверка после обновления памяти

Перед сохранением проверить:

- ID персонажей стабильны;
- новые NPC не дублируют старых;
- скрытый лор не попал публично;
- персонажи знают только то, что могли узнать;
- предметы существуют;
- локации не противоречат карте;
- время движется логично;
- отношения не скачут без причины;
- кто знает чьи имена;
- кто был виден, но не представлен;
- `relationships`, а не `relationship_state`.

## Запрещено

- Не добавлять персонажам знания из hidden lore автоматически.
- Не создавать NPC без ID, если он важен.
- Не менять ID после фиксации.
- Не вводить уже появившегося NPC как нового.
- Не обнулять знакомство, если Акира уже видела/слышала/знает NPC.
- Не делать все слухи общеизвестными сразу.
- Не забывать последствия публичных сцен.
- Не стирать ревность, обиды, слухи и репутацию без причины.
- Не делать Акиру пассивным объектом отношений.
- Не делать Райдену философские выводы вместо поведения.
