# Lock: после хода обязательно применять изменения в state

Этот файл имеет высокий приоритет для каждого игрового хода.

## Главная проблема

Текст сцены не является сохранённым состоянием мира.

`current_state.json` тоже не заменяет остальные state-файлы.

Если после сцены изменились отношения, знания, слухи, репутация, инвентарь, текущая сцена, открытые нитки или состояние персонажей, эти изменения нужно применить в соответствующие файлы `state/*.json`.

Нельзя считать изменения сохранёнными, если они остались только:

- в тексте сцены;
- в ответе GPT;
- в `current_state.json`;
- в `scene_text`;
- в `turn_results` без применения;
- в памяти чата.

Если сцена изменила отношения/репутацию/знания, но нужный state-файл не обновлён, ход считается технически незавершённым.

## Обязательная цепочка после каждой сцены

После написания сцены ИИ обязан:

1. Определить, что реально изменилось.
2. Разложить изменения по конкретным state-файлам.
3. Подготовить patch/turn-result.
4. Применить изменения через API:
   - предпочтительно `/api/v1/sessions/{session_id}/apply-turn-result`, если доступен;
   - или прямым `PUT /api/v1/sessions/{session_id}/json/{file_path}`, если нужно точечно обновить файл.
5. Убедиться, что изменились именно нужные файлы.
6. Только после этого считать ход завершённым.

Если изменений нет — честно указать, что state не менялся.

Если инструмент записи недоступен — не делать вид, что состояние сохранено. Нужно прямо сообщить: `state не обновлён, потому что запись недоступна`.

## Не сохранять только current_state

`state/current_state.json` хранит текущую сцену, место, время, active/nearby персонажей и краткий якорь.

Он НЕ должен быть единственным местом, куда попали последствия сцены.

Если в сцене были:

- личная реакция;
- рост интереса;
- уважение;
- ревность;
- конфликт;
- обида;
- флирт;
- обещание;
- договорённость;
- социальное внимание;
- слух;
- новое знание;
- изменение репутации;
- заметка инструктора;
- будущий крючок;

то нужно обновить профильный state-файл, а не только `current_state.json`.

## Какие файлы обновлять

### Отношения

Если изменилась личная динамика между персонажами, обновить:

`state/relationships.json`

Пары писать в формате:

`character_a__character_b`

Примеры:

- `akira__livia_cross`
- `akira__haru_foster`
- `akira__raiden_sterling`
- `livia_cross__haru_foster`
- `livia_cross__raiden_sterling`
- `raiden_sterling__haru_foster`

Если пары ещё нет, создать её с базовыми шкалами:

```json
{
  "affection": 0,
  "trust": 0,
  "tension": 0,
  "jealousy": 0,
  "respect": 0,
  "curiosity": 0,
  "resentment": 0,
  "status": "краткое состояние отношений после сцены",
  "notes": [],
  "memory": [],
  "open_threads": [],
  "behavior_next": [],
  "triggers": [],
  "last_interaction": null
}
```

### Relationship memory

Для важных пар нельзя сохранять только цифры.

Если сцена дала новый оттенок отношений, обновить также:

- `status` — короткая формула текущей динамики;
- `notes` — важные факты сцены;
- `memory` — что персонажи запомнили друг о друге;
- `open_threads` — что должно вернуться позже;
- `behavior_next` — как это должно влиять на следующую похожую сцену;
- `triggers` — что может изменить динамику;
- `last_interaction` — где/когда/чем закончился последний важный контакт.

Пример:

```json
{
  "pair": "akira__haru_foster",
  "curiosity_delta": 2,
  "respect_delta": 1,
  "status": "Хару заинтересован спокойной колкостью Акиры и тем, что она не играет в обычный флирт.",
  "memory": [
    "Хару запомнил, что Акира у спортивной площадки не суетилась и реагировала коротко.",
    "Акира редким весельем показала, что баскетбол её оживляет."
  ],
  "open_threads": [
    "Хару может вернуться к теме баскетбола как к безопасному поводу снова подойти."
  ],
  "behavior_next": [
    "Хару не должен начинать контакт с нуля; он уже знает, что Акиру цепляет не прямой флирт, а действие и игра."
  ],
  "triggers": [
    "Уважение Хару вырастет, если он увидит контроль Акиры в тренировке или игре."
  ],
  "last_interaction": "вечерняя спортивная зона: баскетбол/наблюдение/короткие реакции"
}
```

## Как менять значения

Изменения должны быть постепенными.

Один взгляд, короткая фраза или случайная встреча не дают большой скачок.

Ориентиры:

- лёгкий интерес: `+1..+3 curiosity`;
- заметный интерес после сцены: `+4..+8 curiosity`;
- маленькое уважение: `+1..+3 respect`;
- явное уважение после действия: `+4..+7 respect`;
- лёгкая привязанность: `+1..+3 affection`;
- маленькое доверие: `+1..+3 trust`;
- лёгкое напряжение: `+1..+4 tension`;
- публичный конфликт: `+5..+12 tension`;
- ревность: `+1..+6 jealousy`;
- болезненная сцена/унижение: `+5..+12 resentment` или `tension`.

Не делать резких скачков без большой сцены.

## Knowledge state

Если персонаж узнал новый факт, обновить:

`state/knowledge_state.json`

Примеры:

- Ливия знает, что Акира оживает на баскетболе.
- Хару услышал имя Акиры.
- Райден заметил, что Акира не реагирует на давление как остальные.
- Инструктор видел необычный энергетический отклик.

Не записывать hidden lore как знание персонажа, если он не узнал это в сцене.

## Reputation state

Если изменилось отношение группы/академии к персонажу, обновить:

`state/reputation_state.json`

Обновлять, если:

- кто-то наблюдал сцену;
- была публичная игра/проверка/конфликт;
- появился повод для слуха;
- инструкторы сделали отметку;
- круг Хару/Райдена/Ливии заметил поведение;
- репутация Акиры стала чуть более конкретной.

Примеры:

- Акиру начали считать странно спокойной.
- У инструкторов появилась отметка по контролю/реакции.
- Хару снова собрал внимание вокруг себя.
- Круг Райдена заметил, что он задержал взгляд на Акире.
- Ливию начали воспринимать как шумную, социальную и тянущуюся к заметным старшим.

## Rumors state

Если появился слух, обновить:

`state/rumors_state.json`

Слух должен иметь источник/причину, даже если она приблизительная.

## Inventory state

Если предмет появился, исчез, был передан, спрятан или раскрыт, обновить:

`state/inventory_state.json`

Не добавлять предметы только в текст сцены без записи в state.

## Current state

Если изменились дата, время, место, active/nearby персонажи, состояние Акиры или цель сцены, обновить:

`state/current_state.json`

Особенно важно обновлять:

- `current_location_id`;
- `time_of_day`;
- `active_characters`;
- `nearby_characters`;
- `current_scene_goal`;
- `visible_inventory`;
- `nearby_items`;
- `last_scene_anchor`.

## Scene history и open threads

Если в сцене было что-то, что должно вернуться позже, обновить:

- `state/scene_history.json`, если используется как state-файл;
- `scene_history.json`, если API хранит историю в корне сессии;
- `open_threads` в `current_state.json` или relationship memory.

Нельзя держать договорённость только в тексте сцены.

Примеры открытых ниток:

- баскетбол через неделю;
- повторная проверка;
- наблюдение Райдена;
- ревность Ливии;
- интерес Хару;
- слух, который должен дойти до круга персонажа.

## Что запрещено

Запрещено:

- писать “отношения изменились”, но не обновлять `relationships.json`;
- обновлять только `current_state.json`, если изменились отношения/репутация/знания;
- оставлять изменения только в `turn_results`;
- сохранять крупные эмоциональные сдвиги без конкретной сцены;
- добавлять знание персонажу, если он не мог его получить;
- удалять старые notes/memory/open_threads без причины;
- перезаписывать весь файл пустым/урезанным состоянием;
- менять future locks как уже случившийся факт;
- писать в отчёте “состояние обновлено”, если API-запись не была выполнена.

## Проверка перед финальным ответом

Перед финальным ответом после сцены проверить:

- были ли реальные изменения отношений;
- были ли изменения репутации;
- были ли новые знания персонажей;
- были ли слухи;
- появились ли открытые нитки;
- какие state-файлы должны быть обновлены;
- были ли они сохранены через API.

Если отношения/репутация/знания изменились, но соответствующий state-файл не обновлён — ход считается незавершённым.

Если запись недоступна, нужно прямо сказать об этом, а не продолжать вид, что память сохранена.
