# Lock: после хода обязательно применять изменения в state

Этот файл имеет высокий приоритет для каждого игрового хода.

## Главная проблема

`turn_results` сам по себе не является сохранённым состоянием мира.

Если после сцены изменились отношения, знания, слухи, репутация, инвентарь, текущая сцена или состояние персонажей, эти изменения нужно применить в соответствующие файлы `state/*.json`.

Нельзя считать изменения сохранёнными, если они остались только в тексте сцены или в `turn_results`.

## Обязательная цепочка после сцены

После написания сцены ИИ обязан:

1. Определить, что реально изменилось.
2. Разложить изменения по state-файлам.
3. Прочитать текущий state-файл через API.
4. Аккуратно обновить только нужные поля.
5. Сохранить файл через API `PUT /api/v1/sessions/{session_id}/json/{file_path}`.
6. При необходимости перечитать файл и убедиться, что изменения появились.

Если нет изменений — честно указать, что state не менялся.

## Отношения

Если в сцене изменились отношения между персонажами, обновить:

`state/relationships.json`

Пары писать в формате:

`character_a__character_b`

Пример:

`akira__haru_foster`

`akira__raiden_sterling`

`livia_cross__haru_foster`

Если пары ещё нет, создать её с базовыми шкалами:

```json
{
  "affection": 0,
  "trust": 0,
  "tension": 0,
  "jealousy": 0,
  "respect": 0,
  "curiosity": 0,
  "resentment": 0,
  "status": "краткое состояние отношений после сцены",
  "notes": []
}
```

## Как менять значения

Изменения должны быть постепенными.

Один взгляд, короткая фраза или случайная встреча не дают большой скачок.

Ориентиры:

- лёгкий интерес: +1..+3 curiosity;
- заметный интерес после сцены: +4..+8 curiosity;
- маленькое уважение: +1..+3 respect;
- явное уважение после действия: +4..+7 respect;
- лёгкое напряжение: +1..+4 tension;
- публичный конфликт: +5..+12 tension;
- ревность: +1..+6 jealousy;
- болезненная сцена/унижение: +5..+12 resentment или tension;
- доверие растёт медленнее всего: обычно +1..+3 за сцену.

Не делать резких скачков без большой сцены.

## Notes

В `notes` добавлять только важные сценовые факты, которые должны сохраниться.

Правильно:

- `Хару заметил, что обычный флирт на Акиру почти не действует.`
- `Ливия болезненно отреагировала на внимание Хару к Акире.`
- `Райден резко отреагировал на навязчивое прикосновение Киары.`

Неправильно:

- `Они посмотрели друг на друга.`
- `Была сцена в столовой.`

## Knowledge state

Если персонаж узнал новый факт, обновить:

`state/knowledge_state.json`

Примеры знания:

- Ливия знает, что Акира использовала пространственный канал для дыма.
- Хару услышал имя Акиры.
- Райден заметил, что Акира не реагирует на давление как остальные.
- Вероника увидела, что Акиру сложно втянуть в публичное соревнование.

Не записывать hidden lore как знание персонажа, если он не узнал это в сцене.

## Reputation state

Если изменилось отношение группы/академии к персонажу, обновить:

`state/reputation_state.json`

Примеры:

- слух о белых волосах Акиры усилился;
- Акиру начали считать странно спокойной;
- Вероника публично проиграла темп;
- Хару снова собрал внимание вокруг себя.

## Rumors state

Если появился слух, обновить:

`state/rumors_state.json`

Слух должен иметь источник/причину, даже если она приблизительная.

## Inventory state

Если предмет появился, исчез, был передан, спрятан или раскрыт, обновить:

`state/inventory_state.json`

Не добавлять предметы только в текст сцены без записи в state.

## Current state

Если изменились дата, время, место, активные/nearby персонажи, состояние Акиры или цель сцены, обновить:

`state/current_state.json`

Особенно важно обновлять:

- `current_location_id`;
- `time_of_day`;
- `active_characters`;
- `nearby_characters`;
- `current_scene_goal`;
- `visible_inventory`;
- `nearby_items`.

## Что запрещено

Запрещено:

- писать “отношения изменились”, но не обновлять `relationships.json`;
- оставлять изменения только в `turn_results`;
- сохранять крупные эмоциональные сдвиги без конкретной сцены;
- добавлять знание персонажу, если он не мог его получить;
- удалять старые notes без причины;
- перезаписывать весь файл пустым/урезанным состоянием;
- менять future locks как уже случившийся факт.

## Проверка перед финальным ответом

Перед финальным ответом после сцены проверить:

- были ли реальные изменения;
- какие state-файлы должны быть обновлены;
- были ли они сохранены через API;
- если нет — коротко сообщить, что state не изменялся.

Если отношения изменились, но `relationships.json` не обновлён — ход считается незавершённым.
