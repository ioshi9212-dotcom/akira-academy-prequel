# Lock: gameplay visible scene before state and no status summary

Этот lock не добавляет новый характер персонажам. Он фиксирует порядок runtime: сначала видимая сцена, потом сохранение state, после tool call — возврат именно сцены.

## 1. Сбор данных перед сценой

Перед рендером сцены GPT должен собрать runtime-пакет из уже загруженных источников:

- `current_state`;
- active/nearby персонажи;
- character runtime slices для присутствующих персонажей;
- relationship slice;
- knowledge slice;
- inventory / visible items;
- scene goal;
- direct player input parse;
- stop conditions.

Этот пакет внутренний. Его нельзя показывать пользователю в игровом режиме.

## 2. Сначала `visible_scene_text`

В игровом режиме сначала нужно полностью собрать `visible_scene_text`.

`visible_scene_text` — это полный видимый ответ игроку:

- шапка сцены;
- тело сцены;
- точные реплики Акиры из текущего ввода игрока, если они есть;
- действия из скобок как действие/паузу/движение;
- реакции NPC/мира;
- последствие, наблюдение, конфликт, крючок или продвижение;
- нижние блоки `Что можно сделать`, `Что Акира могла бы сказать`, `Мысли Акиры`.

Нельзя считать игровой ход завершённым, если пользователь не увидел `visible_scene_text`.

## 3. State не заменяет сцену

`apply-turn-result`, `PUT json`, `save`, `dry_run` и любые tool calls являются только техническим сохранением.

Они не являются игровым ответом.

После любого tool call финальный ответ в gameplay mode всё равно обязан быть сценой, а не отчётом.

## 4. Структура apply-turn-result

Если состояние нужно сохранить через `apply-turn-result`, GPT должен вызвать его только после того, как `visible_scene_text` уже собран.

В request нужно передать:

```json
{
  "data": {
    "current_state_changes": {},
    "story_lines_changes": {},
    "relationship_changes": [],
    "knowledge_changes": {},
    "reputation_changes": {},
    "rumors_changes": {},
    "inventory_changes": {}
  },
  "visible_scene_text": "ПОЛНАЯ ВИДИМАЯ СЦЕНА С ШАПКОЙ, ТЕЛОМ И НИЖНИМИ БЛОКАМИ"
}
```

Если дополнительно нужен внутренний пакет, можно передать `render_packet`, но не показывать его пользователю.

## 5. После apply-turn-result

После ответа tool финальный видимый ответ должен быть:

```txt
final_scene_text
```

или, если tool вернул только `visible_scene_text`:

```txt
visible_scene_text
```

Ответы по `status`, `changed_files`, `source`, `dry_run` — внутренние и не должны становиться финальным сообщением в gameplay mode.

## 6. Запрет summary/status вместо сцены

В gameplay mode запрещены финальные ответы вида:

- `Сцена отработана`;
- `Ключевые моменты`;
- `Следующая точка`;
- `Состояние обновлено`;
- `Ход сохранён`;
- `Браслет и маршрутный лист отмечены` как статус вместо сцены;
- `Если хочешь, могу отрендерить`;
- любой список итогов вместо видимого рендера сцены.

Если GPT начал писать такой ответ, он обязан заменить его на `final_scene_text/visible_scene_text` до отправки.

## 7. Repair-render

Если `apply-turn-result` был ошибочно вызван до видимого рендера сцены, следующий ответ должен быть repair-render.

Repair-render:

- показывает уже сохранённое событие как полноценную сцену;
- не меняет state повторно;
- не вызывает `apply-turn-result` второй раз;
- не оправдывается и не показывает технический отчёт;
- восстанавливает соответствие между видимым текстом и памятью.

## 8. No Akira dialogue without player speech

Если текущий ввод игрока не содержит текста вне скобок, в теле сцены запрещено создавать строку:

```txt
**Акира** — ...
```

Акира может действовать, смотреть, молчать, двигаться, физически реагировать или держать паузу.

Возможные фразы Акиры идут только в нижний блок `Что Акира могла бы сказать`.
